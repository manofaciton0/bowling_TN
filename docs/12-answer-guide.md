# 12. 확인 문제 핵심 해설

이 문서는 각 장의 확인 문제에 대한 핵심 방향을 제공한다. 문장을 외우기보다 자신의 말로 다시 설명하는 것이 중요하다.

## 00. 전체 구조

1. **HTML에 팀 카드를 미리 쓰지 않는 이유**: 팀, 경기, 대회 형식이 데이터에 따라 달라지고 같은 상태에서 모바일과 데스크톱 화면을 모두 만들기 위해서다.
2. **상태를 먼저 바꾸는 이유**: 화면을 상태의 결과로 유지하면 데이터와 DOM의 불일치를 줄일 수 있다.
3. **저장소 역할**: localStorage는 현재 기기의 빠른 캐시와 복구, Supabase는 여러 기기가 공유하는 원격 상태다.
4. **전체 재렌더링**: 구현과 일관성은 단순하지만 포커스가 사라지고 불필요한 DOM 작업이 발생한다.

## 01. HTML과 DOM

1. `createElement()`로 만든 요소는 `appendChild()` 등으로 문서에 연결될 때 보인다.
2. `textContent`는 문자열을 HTML로 실행하지 않아 삽입 공격 위험이 낮다.
3. 요소가 없을 수 있으므로 `?.`로 안전하게 이벤트 연결을 건너뛴다.
4. JavaScript는 의미, CSS는 표현을 담당해 테마와 상태 스타일 변경이 쉬워진다.

## 02. CSS와 반응형

1. CSS 변수는 같은 역할의 색상을 한 곳에서 바꿀 수 있게 한다.
2. 넓은 연결선 대진표를 축소하면 글자와 터치 영역이 너무 작고 브라우저 줌에 취약하다.
3. Flex 자식의 긴 텍스트를 줄이고 말줄임해야 할 때 `min-width: 0`이 필요하다.
4. Safari 확대 방지를 위해 입력 글자를 16px 이상으로 둔다.

## 03. 상태와 JSON

1. `bracketState[2][1]`은 3번째 라운드의 2번째 경기다.
2. `winner: null`은 승자가 없고 `p1Score: ""`는 점수 문자열이 비어 있다는 뜻이다.
3. 원격 JSON을 다시 받으면 객체 참조가 달라질 수 있어 안정적인 식별값으로 비교해야 한다.
4. 일반 경기보다 슬롯이 많은 결승을 같은 함수로 순회하기 위해 `slots`가 필요하다.
5. 로컬 형식 버전은 구조 호환성을, DB 숫자 버전은 변경 순서와 동시성 판단을 돕는다.

## 04. 렌더링

1. 인덱스는 클릭 시 원래 `bracketState` 위치를 찾고 `data-*` 속성을 만들 때 필요하다.
2. 조기 반환 후 나머지 코드는 팀 객체가 존재한다고 가정할 수 있다.
3. 관리자만 클릭 콜백과 점수 편집기를 활성화한다.
4. 상태와 View가 분리되어 있어 같은 데이터에서 서로 다른 DOM 구조를 만들 수 있다.
5. 전체 렌더링은 기존 입력 요소 자체를 DOM에서 제거하기 때문에 포커스와 팝오버가 사라진다.

## 05. 이벤트와 점수

1. `handler()`는 이벤트 등록 시 즉시 실행된 결과를 전달한다. 함수 자체인 `handler`를 전달해야 한다.
2. 입력 클릭이 부모 팀 카드 클릭으로 전파되어 승자 선택까지 실행되는 것을 막는다.
3. 원본 점수 배열을 제출 전에 변경하지 않기 위해 복사한다.
4. 요청 수와 재렌더링을 줄이고 취소 가능한 임시 입력을 제공하기 위해서다.
5. `reduce()`는 개인 점수 배열을 팀 합계 하나로 만든다.

## 06. 토너먼트 로직

1. 이전 승자가 다음 라운드에서 만든 승자와 점수는 새 승자에게 유효하지 않기 때문이다.
2. 목적 슬롯이 `p1`, `p2`, `p3`처럼 데이터로 주어지므로 `nextMatch[route.slot]`이 필요하다.
3. 동점 상태에서 임의로 한 팀을 선택하면 규칙과 데이터가 불일치한다.
4. 4등은 본선 6강 패배팀 점수만으로 결정되므로 결승 승자가 없어도 계산할 수 있다.
5. 파생 상태를 저장하지 않으면 원본 변경 시 순위 불일치가 줄어든다.

## 07. localStorage

1. localStorage는 문자열만 저장하므로 객체를 `JSON.stringify()`해야 한다.
2. 다른 대진 구조의 오래된 상태를 잘못 불러오는 것을 막는다.
3. 원격 실패 전에도 현재 기기의 입력을 빠르게 보존할 수 있다.
4. 1초 간격은 350ms 범위를 벗어나므로 네 번 저장된다.
5. 외부 파일은 누락되거나 조작될 수 있어 구조와 버전을 확인해야 한다.

## 08. Supabase Auth

1. Publishable Key는 공개 클라이언트 식별용이며 실제 데이터 권한은 JWT와 RLS가 결정한다.
2. 인증된 일반 사용자와 실제 대회 관리자를 분리하기 위해서다.
3. 사용자는 브라우저 CSS와 JavaScript를 바꿀 수 있다.
4. `auth.uid()`는 현재 요청 JWT의 사용자 UUID다.
5. 결과가 없거나 정확히 하나여야 하는 UUID 멤버십 조회에 적합하다.

## 09. 데이터와 Realtime

1. 현재 상태를 조회하는 동안 발생한 변경 이벤트를 놓칠 가능성을 줄인다.
2. Supabase SDK가 JSONB 응답을 JavaScript 객체로 변환한다.
3. `players`, `bracket_state`, `display_state`가 포함된다.
4. RLS는 SQL 동작별로 SELECT와 UPDATE 정책을 따로 평가한다.
5. 요청 중 새 저장이 동시에 실행되는 것을 줄이고 종료 후 최신 변경을 다시 저장하기 위해서다.

## 10. 동시성과 디버깅

1. debounce는 가까운 시간의 요청만 합칠 뿐 서로 다른 기기의 stale state를 막지 못한다.
2. 현재 버전 검사는 주로 들어오는 Realtime 이벤트 순서에 적용된다.
3. 다른 클라이언트가 먼저 저장해 DB 버전이 달라졌음을 뜻한다.
4. 서로 다른 경기와 점수가 서로 다른 행이면 한 행 전체를 덮어쓰지 않아도 된다.
5. 사용자 확대 기능을 막으면 접근성이 나빠지므로 입력 글자 크기 원인을 해결한다.

## 실습 해답을 검토하는 기준

정답 코드는 한 가지가 아니다. 다음 기준을 만족하는지 확인한다.

```text
데이터와 DOM이 분리되어 있는가?
상태 변경 경로가 함수로 모여 있는가?
승자 변경 시 파생 상태를 정리하는가?
입력 중 원본을 불필요하게 변경하지 않는가?
외부 데이터의 형식을 검증하는가?
프론트엔드 권한과 DB 권한을 구분하는가?
동시 수정 실패를 조용히 무시하지 않는가?
```
